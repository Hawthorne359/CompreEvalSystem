# 评审阈值与任务流转现状基线（改造前）

## 1. 分差阈值当前执行逻辑

- 配置来源：`eval.ReviewRule.score_diff_threshold`（单一阈值）。
- 触发入口：`scoring/views.py` 的常规评分提交与仲裁接口。
- 比较对象：**同一提交 + 同一指标**的常规评分记录（排除仲裁轮 `round_type=3`）。
- 判定方式：`max(scores) - min(scores) > score_diff_threshold`。
- 结论：
  - 不是比“总分差值”。
  - 不是按模块分别阈值。
  - 当前仅支持“单指标统一阈值”。

## 2. 评审任务当前流转逻辑

- 任务生成入口：`POST /api/v1/review/assignments/generate/`。
- 任务模型：`scoring.ReviewAssignment`，状态仅有 `assigned/completed/cancelled`。
- 生成行为（当前）：
  - 双评开启：直接给助理生成 `assigned` 任务（必要时加辅导员确认任务）。
  - 单评模式：直接给辅导员或助理生成 `assigned` 任务。
- 结论：
  - 目前没有“先到辅导员池，再放行给助理”的中间状态。
  - 任务是直接落到最终执行人。

## 3. UI 当前语义问题

- 页面：`frontend/src/views/ProjectConfig.vue`
- 问题：按钮文案固定为“生成双评任务”，即使已关闭双评仍不切换语义。

## 4. 兼容改造基线

- 阈值升级方向：新增“模块阈值 + 总分阈值”，并保留旧字段回退能力。
- 流转升级方向：引入辅导员池中间态（如 `pending_counselor_release`），避免破坏现有 `assigned` 消费链路。
- UI 升级方向：按 `dual_review_enabled` 联动按钮名称与说明文案。
