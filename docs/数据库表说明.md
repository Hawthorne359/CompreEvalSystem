# 数据库表说明

> 生成日期：2026-03-04  
> 数据库共 **30 张表**，分为：业务表（22）、Django 基础设施表（8）。  
> 结论先行：**无遗留死表**，`audit_audit_log` 已在迭代中被正式删除；  
> `users_user_groups` / `users_user_user_permissions` 虽无业务数据，但属 Django 框架必需，不可删除。

## 一、业务表（22 张）

### 1. 认证与用户模块（3 张）

#### `users_user`
**对应模型**：`users.User`（继承自 Django `AbstractUser`）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BigAutoField | 主键 |
| username | VARCHAR(150) | 登录用户名，唯一 |
| password | VARCHAR(128) | 加密密码（bcrypt） |
| name | VARCHAR(150) | 完整姓名 |
| email | VARCHAR(254) | 邮箱 |
| is_active / is_staff / is_superuser | Boolean | Django 内置权限标志 |
| phone | VARCHAR(20) | 手机号 |
| employee_no | VARCHAR(50) | 工号，唯一（教职工） |
| student_no | VARCHAR(50) | 学号，唯一（学生） |
| department_id | FK → org_department | 所属院系（学生无、教职工有） |
| class_id | FK → org_class | 所属班级（仅学生） |
| current_role_id | FK → users_role | 当前激活角色，用于前端菜单/权限渲染 |
| date_joined / last_login | DateTime | 注册时间 / 最近登录 |
| created_at / updated_at | DateTime | 记录创建/更新时间 |

> **说明**：系统所有角色（学生/辅导员/评审教师/主任/管理员/超管）均在此表存储，通过 `users_user_role` 关联具体身份。

---

#### `users_role`
**对应模型**：`users.Role`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | AutoField | 主键 |
| name | VARCHAR(50) | 角色显示名称（如"主任"） |
| code | VARCHAR(50) | 角色唯一标识符（如 `director`），唯一 |
| level | SmallInt | 权限层级，0 最低；数字越大权限越高 |
| description | Text | 角色描述 |
| created_at | DateTime | 创建时间 |

> **层级定义（当前）**：0=学生 / 1=辅导员 / 2=评审教师 / 3=主任 / 4=管理员 / 5=超级管理员  
> 分配某角色时，系统会自动为用户补全该 level 及以下所有角色（`users_user_role` 信号机制），用于前端角色切换下拉。

---

#### `users_user_role`
**对应模型**：`users.UserRole`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | AutoField | 主键 |
| user_id | FK → users_user | 所属用户 |
| role_id | FK → users_role | 关联角色 |
| scope_type | VARCHAR(50) | 作用域类型（如 `class` 表示负责某班级，空串表示全局） |
| scope_id | PositiveInt | 作用域 ID（如班级 ID），与 scope_type 配合 |
| is_primary | Boolean | 是否为主身份（目前未大量使用） |
| created_at | DateTime | 创建时间 |

> **唯一约束**：`(user_id, role_id, scope_id, scope_type)`  
> **重要设计**：同一个用户的同一角色可能有多条记录——一条 `scope_type=''`（基础角色）+ N 条 `scope_type='class'`（负责的每个班级各一条）。前端角色切换下拉已按 `role.id` 去重展示。

### 7. 审计与补交模块（5 张）

#### `audit_operation_log`
**对应模型**：`audit.OperationLog`

系统核心日志表，**所有角色的所有写操作均在此留痕**。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | AutoField | 主键 |
| user_id | FK → users_user（SET_NULL） | 操作者（账号删除后置空） |
| username_snapshot | VARCHAR(150) | 操作时的用户名快照（账号删除后仍可追溯） |
| role_snapshot | VARCHAR(50) | 操作时的角色名称快照 |
| ip_address | GenericIPAddressField | 来源 IP（由中间件自动采集） |
| user_agent | VARCHAR(500) | 浏览器标识 |
| module | VARCHAR(20) | 业务模块：`auth`/`users`/`org`/`eval`/`submission`/`scoring`/`appeal`/`report`/`system` |
| action | VARCHAR(100) | 操作码（如 `login`/`create_season`/`delete_project`） |
| level | VARCHAR(10) | 日志等级：`INFO`（一般）/`NOTICE`（关键）/`WARNING`（敏感）/`CRITICAL`（高危） |
| target_type | VARCHAR(100) | 被操作对象类型（如 `eval_season`） |
| target_id | PositiveInt | 被操作对象 ID |
| target_repr | VARCHAR(500) | 被操作对象的可读名称（如"2024-2025学年春季"） |

| reason | Text | 操作理由（WARNING/CRITICAL 级必填） |

| created_at | DateTime | 精确到秒的操作时间（自动填充） |

> **索引**：`(user_id, created_at)` / `(module, action)` / `(level, created_at)` / `(is_audit_event, created_at)` / `(is_abnormal, created_at)` / `created_at`，支持各维度高效检索。
