# 双评策略回归矩阵与灰度上线清单

## 一、回归测试矩阵（必须执行）

### 1. 角色与权限
- 超级管理员可配置项目评审规则、可触发任务分配生成。
- 院系主任仅在本院系范围可改评审规则，越权项目被拒绝。
- 评审老师仅在自身管辖范围可改评审规则，越权被拒绝。
- 学生助理只能评分“被分配任务”，未分配提交必须 403。
- 评审老师评分接口同样必须命中任务分配，未命中必须 403。

### 2. 双评策略行为（same_counselor_classes）
- 配置 `review_scope_mode=same_counselor_classes` 后，生成任务成功。
- A/B 两个班在同一辅导员下可互评（跨班）。
- 开启 `cross_class_shuffle_enabled` 后，多次生成版本，分配对象可打散变化。
- 关闭打散后，分配结果稳定可复现（同样本同版本规则）。

### 3. 数据流与状态
- 学生提交 `draft -> submitted -> under_review -> approved/rejected` 正常。
- 评分记录写入后 `final_score` 正常重算，仲裁后结果覆盖逻辑正常。
- 项目评定截止后，评分提交被拒绝（补交通道特例除外）。
- 评审任务列表和助理任务列表展示“最新版本分配任务”。

### 4. 兼容与降级
- 未生成任何分配任务的历史项目：评审列表可读，写入被策略约束逻辑正确处理。
- 旧项目评审规则无新字段时，默认值生效不报错。

## 二、测试数据模板（建议）
- 1 个 ongoing 项目（本学年中测）。
- 1 名辅导员，负责 A/B 两班。
- 每班 2 名学生，至少 1 名学生助理。
- 至少 4 份提交（每班各 2 份），其中 1 份人工制造分差触发仲裁。

## 三、灰度上线步骤
1. **预发布验证**
   - 执行迁移：`eval.0008`、`scoring.0003`。
   - 执行 `manage.py check` 与前端 `npm run build`。
2. **小范围开关**
   - 先选 1 个项目启用 `same_counselor_classes`。
   - 仅允许超管配置，院系主任/评审老师权限先观察审计日志。
3. **观察期（1-2 天）**
   - 监控评分提交 403 比例（排查误拒绝）。
   - 监控分配生成日志与评分日志一致性。
4. **逐步放开**
   - 放开更多项目使用跨班互评。
   - 放开院系主任、评审老师配置权限。

## 四、回滚开关与应急策略

### 回滚开关（配置层）
- 将 `review_scope_mode` 统一改回 `same_class`。
- 暂停“生成双评任务”操作，仅保留已有评分读取。

### 应急降级（服务层）
- 若分配服务异常：
  - 临时禁止评分写入（保留查看），防止错误评分污染。
  - 锁定 `assignment_version`，禁止重算新版本。

### 数据安全
- 已存在 `ScoreRecord`/`ArbitrationRecord` 不做删除。
- 回滚仅影响“后续任务分配策略”，不破坏历史分数可追溯性。

## 五、上线验收门槛
- 权限越权用例全部通过（0 漏放行）。
- 双评任务分配可追踪（项目、版本、评审人、轮次完整）。
- 关键主流程（学生提交->双评->仲裁->出分）通过率 100%。
