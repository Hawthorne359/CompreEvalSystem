# 核心流程主链路审计清单（中测优先）

## 审计范围
- 超管配置测评周期/项目并开启
- 学生按题目化完成作答并提交
- 辅导员指派学生助理
- 助理/评审老师完成双评，必要时仲裁并产出最终分

## 当前已可跑通的部分
- **项目创建与时间控制**：已有周期/项目/评审规则完整配置入口，状态与时间窗口校验已具备。
- **学生提交流程**：题目化作答、附件上传、提交前完整性校验已具备。
- **助理指派能力**：辅导员可按班级指派与撤销学生助理，具备范围校验和日志。
- **评分与仲裁**：已有双评、仲裁、总分重算主流程，最终分计算逻辑稳定。

## 核心卡点（需优先修复）
1. **双评“谁来评”缺少任务分配模型**
   - 现状：评分接口以“是否已有他人评分”推断轮次，缺少明确任务归属。
   - 风险：容易出现未分配用户越权评分、同人重复评分、跨班策略无法落地。
2. **双评策略配置缺字段**
   - 现状：`ReviewRule` 只有开关/阈值/最终分规则，没有“范围模式、打散、每份分配人数”。
   - 风险：无法实现“同辅导员跨班互评”。
3. **评审规则权限未按新要求下放**
   - 现状：评审规则更新仍按管理员口径判断。
   - 风险：院系主任/评审老师无法在权限范围内配置策略。
4. **前端审核详情页角色判断仍存在旧 code 依赖**
   - 现状：页面中仍按 `reviewer/admin` 等 code 分支控制按钮。
   - 风险：与等级权限逻辑不一致，导致功能可见性异常。
5. **助理任务列表仍按班级范围泛化，不是“被分配任务”**
   - 现状：助理看到的是 scope 班级下全部提交，而非精确分配任务。
   - 风险：策略落地后会出现“看得见但不该评”的数据。

## 立即落地策略
- 先新增“评审任务分配”模型并写入任务归属，再在评分入口强制任务校验。
- 双评策略首期只开 `same_counselor_classes`，并支持打散。
- 保留兼容兜底：历史项目无分配任务时，只允许读，不允许写评分。

## 验收样本（最小可跑）
- 1 个 ongoing 项目
- 1 名辅导员，管辖 2 个班（A/B）
- 每班至少 2 名学生，A/B 各至少 1 名助理
- 至少 2 份提交触发双评，其中 1 份触发仲裁

## 通过标准
- 评审任务可追踪（谁、评哪份、第几轮）
- 未分配用户评分被拒绝
- 同辅导员跨班互评可执行
- 仲裁后最终分可重算并正确落库
