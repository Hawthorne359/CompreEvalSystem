# 校园综合测评管理系统 — 按模块逐步完善指南

## 使用说明（给后续 AI 或开发者）

- **当前状态**：后端 API 已基本实现（Django + DRF），前端核心页面已落地并持续迭代中；本文档用于对齐接口与模块边界，避免重复开发和错误回退。
- **本文档**：按**模块**拆分，每个模块独立成节。做某一模块时，只需打开对应一节，按「后端接口清单」「前端要做的」「设计要点」「交付标准」实施即可；无需通读全文。
- **建议顺序**：按下面模块编号依次做（0→1→2→…），因为后面的模块会依赖前面（如学生提交依赖「测评项目」已存在，审核依赖「提交」）。
- **给 AI 的用法**：复制「本节交给 AI 的提示语」整段 + 本模块那一节内容，在新对话里粘贴，即可让 AI 只做这一块并正确对接。

---

## 项目技术栈与约定（全局）

- **后端**：Django 6 + DRF，JWT 认证，基础路径 `http://127.0.0.1:8000/api/v1/`。所有接口除登录/刷新外需在请求头带 `Authorization: Bearer <access_token>`。
- **前端**：Vue 3 + Vue Router + Pinia + Axios + Tailwind，开发时 `npm run dev` 跑在 `http://localhost:5173`，vite 已代理 `/api` 到后端。请求用 `frontend/src/api/axios.js` 里的 `api` 实例（已带 baseURL 和 token）。
- **毕设要求**：界面**简洁、正式，符合学校管理系统风格**；不花哨，信息层级清晰，表格/表单易读。

# 模块0：登录与布局（可选优化）

**现状**：登录页、Layout 导航、退出已可用；登录后 `/users/me/` 会返回用户与角色；顶部显示当前用户名和当前角色。

**本模块可选做**：
- 角色切换下拉：调用 `POST /api/v1/auth/switch-role/`，body `{ "role_id": number, "password": "" }`（切到 admin/superadmin 时必填 password）。
- 刷新页面后恢复用户信息：在 Layout 或 App 里 onMounted 调 `GET /api/v1/users/me/` 写入 Pinia 的 auth store。

**后端接口（已存在）**：
- `POST /api/v1/auth/login/`  body: `{ username, password }`  → 返回 `{ access, refresh, user }`
- `GET /api/v1/users/me/`  → 返回当前用户（含 user_roles、current_role）

**前端文件**：`frontend/src/views/Login.vue`、`frontend/src/views/Layout.vue`、`frontend/src/stores/auth.js`

# 模块1：组织架构（院系、班级）

**目标**：管理员可查看院系、班级列表；在「系统管理」或单独菜单下展示，供后续「测评项目」「用户管理」等选择院系/班级时使用。默认以展示与筛选为主，增删改可按权限策略择期启用。

**后端接口（已存在）**：
- `GET /api/v1/departments/`  
  - 可选 query：`?tree=1` 树形。  
  - 返回：`[{ id, name, code, parent, ... }]`
- `GET /api/v1/classes/`  
  - 可选 query：`?department=<id>&grade=xxx`  
  - 返回：`[{ id, name, department, department_name, grade, academic_year }]`
- `GET /api/v1/classes/<id>/students/`  
  - 返回：`{ class: {...}, students: [{ id, username, ... }] }`

**前端要做的**：
- 新建页面或在本模块专用页：**院系列表**（表格或树形）；**班级列表**（表格，支持按院系、年级筛选）；点击某班级可展示**该班学生列表**。
- 调用上述三个接口，用 `api.get(...)`，数据渲染到表格/树。
- 若没有单独菜单，可放在「系统管理」下做一个 Tab 或子页「组织架构」。

**设计要点**：表格列清晰（院系名/编码、班级名/院系/年级、学生学号/姓名等）；风格简洁正式。

**交付标准**：管理员登录后能打开该页，看到院系、班级、班级学生列表且数据来自接口。

# 模块2：测评项目与规则配置

**目标**：管理员可管理「测评周期」与「项目」，以及项目的**指标**、**总分权重规则**、**双评/仲裁规则**。学生提交、审核、报表都依赖「项目」存在。

**后端接口（已存在）**：
- `GET /api/v1/seasons/`  → 测评周期列表  
- `POST /api/v1/seasons/`  body: `{ name, academic_year, semester, status?, start_time?, end_time? }`  → 创建周期  
- `GET /api/v1/seasons/<season_id>/projects/`  → 该周期下项目列表  
- `POST /api/v1/seasons/<season_id>/projects/`  body: `{ name, description?, status?, allow_late_submit?, late_submit_deadline? }`  → 创建项目  
- `GET /api/v1/projects/<id>/`  → 项目详情  
- `PATCH /api/v1/projects/<id>/`  body: 同上字段  → 更新项目  
- `GET /api/v1/projects/<project_id>/indicators/`  → 指标列表  
- `POST /api/v1/projects/<project_id>/indicators/`  body: `{ name, category, max_score?, weight?, order? }`  → 创建指标  
- `GET/PATCH/DELETE /api/v1/projects/<project_id>/indicators/<id>/`  → 指标详情/更新/删除  
- `GET /api/v1/projects/<project_id>/weight-rule/`  → 总分权重规则（可能自动创建）  
- `PATCH /api/v1/projects/<project_id>/weight-rule/`  body: `{ formula_type?, formula_config? }` 例如 `formula_config: { "A": 0.3, "B": 0.3, "C": 0.2, "D": 0.2 }`  
- `GET /api/v1/projects/<project_id>/review-rule/`  → 双评/仲裁规则  
- `PATCH /api/v1/projects/<project_id>/review-rule/`  body: `{ dual_review_enabled?, score_diff_threshold?, final_score_rule?, allow_view_other_scores?, require_arbitration_above_threshold? }`  （final_score_rule: `average`|`max`|`first`）

**前端要做的**：
- 页面：**测评周期列表** + 可新建周期；点击某周期进入**该周期下的项目列表** + 可新建项目；点击某项目进入**项目配置**，包含：
  - 项目基本信息（编辑）
  - **指标列表**（表格，增删改）
  - **总分权重规则**（表单：A/B/C/D 等权重，对应 formula_config）
  - **评审规则**（双评开关、分差阈值、最终分取法、是否允许看他人评分等）
- 所有写操作用 `api.post` / `api.patch` / `api.delete`，读操作用 `api.get`。

**设计要点**：分步骤或 Tab（周期 → 项目 → 指标与规则）；表单、表格清晰；权重与规则说明简短易懂。

**交付标准**：管理员能创建周期、项目，为项目配置指标和权重/评审规则，且保存后再次打开能正确回显。

# 模块3：用户管理

**目标**：管理员可查看用户列表、创建用户、编辑用户（含分配院系/班级、角色），以及重置密码。辅导员/主任等需要先有用户并分配角色。

**后端接口（已存在）**：
- `GET /api/v1/users/`  → 用户列表（分页，query 可带 `is_active`, `department`, `class_obj` 等）
- `POST /api/v1/users/`  body: `{ username, password?, email?, name?, phone?, employee_no?, student_no?, department?, class_obj?, is_active? }`  → 创建用户
- `GET /api/v1/users/<id>/`  → 用户详情（含 user_roles）
- `PATCH /api/v1/users/<id>/`  body: 同上（含 password 则改密）  → 更新用户
- `POST /api/v1/users/import/`  **form-data**：`file`（Excel）  → 批量导入用户
- `GET /api/v1/users/import/template/`  → 下载用户导入模板（Excel）

**注意**：角色分配在后端通过 `UserRole` 表维护；若后端未提供「为用户添加/删除角色」的独立接口，可在本模块只做「用户 CRUD + 院系/班级」，角色分配可后续在 Django Admin 或单独接口完成。若已有接口则一并做。

**前端要做的**：
- 页面：**用户列表**（表格，支持按状态/院系/班级筛选、分页）；**新建用户**（表单）；**编辑用户**（表单，含密码可选填）；**批量导入用户**（上传 Excel + 下载模板）。
- 列表和详情调用上述接口；创建/更新用 POST/PATCH；批量导入页面支持模板下载与结果展示。

**设计要点**：表格列包含用户名、姓名、学号/工号、院系、班级、状态等；表单分组清晰；批量导入页面含使用说明与错误详情。

**交付标准**：管理员能列表、新建、编辑用户，批量导入用户功能可用，数据与后端一致。

# 模块4：学生端 - 我的提交

**目标**：学生可看到自己的提交列表（按项目/状态筛选）、新建草稿、编辑自评与佐证、正式提交；可查看每条提交的状态与得分（若已出分）。毕设要求：提交材料 + 自评分数 + 上传佐证；查看审核状态、最终得分、评审意见。

**后端接口（已存在）**：
- `GET /api/v1/submissions/`  query: `?project=<id>&status=xxx`  → 当前用户的提交列表  
- `POST /api/v1/submissions/`  body: `{ project, self_score?, remark? }`  → 创建草稿  
- `GET /api/v1/submissions/<id>/`  → 提交详情（含 evidences、project_name 等）  
- `PATCH /api/v1/submissions/<id>/`  body: `{ self_score?, remark? }`  → 更新草稿（仅 draft）  
- `POST /api/v1/submissions/<id>/submit/`  → 正式提交（将状态改为 submitted）  
- `POST /api/v1/submissions/<id>/evidences/`  **form-data**：`file`（必填）, `category?`, `name?`  → 上传佐证  

**前端要做的**：
- 页面：**我的提交列表**（表格：项目名、状态、提交时间、最终得分等；筛选：项目下拉、状态下拉）；**新建提交**（选择项目、自评分数可 JSON 或按指标填、备注）；**提交详情**（展示基本信息、自评、佐证列表、状态、得分与评审意见；若为草稿可编辑、上传佐证、点击「正式提交」）。
- 佐证上传用 `FormData`，`api.post(url, formData, { headers: { 'Content-Type': 'multipart/form-data' } })`。
- 状态与得分：从详情接口的 `status`、`final_score`、以及后续可从「评分记录」或详情中带出的评审意见展示。

**设计要点**：学生端界面简洁；步骤明确（选项目 → 填自评/传佐证 → 正式提交）；列表与详情信息完整可读。

**交付标准**：学生登录后能完成「选项目→填自评→上传佐证→正式提交」全流程，并能查看状态与得分。

# 模块5：审核与评分

**目标**：辅导员可对待审提交做「初审」；评审教师可对提交的各个指标打分（双评）；分差超阈值时由仲裁教师做仲裁；管理员可 Excel 批量导入成绩。毕设要求：多级审核、双评网阅、阈值与最终分取法可配置、实名打分。

**后端接口（已存在）**：
- `GET /api/v1/review/tasks/`  → 待我审核/已审核的提交列表（辅导员/评审/主任/管理员）  
- `GET /api/v1/review/submissions/<id>/`  → 某条提交的审核视图（含详情、佐证）  
- `POST /api/v1/review/submissions/<id>/initial/`  → 辅导员初审（将状态改为 under_review）  
- `GET /api/v1/review/submissions/<id>/scores/`  → 当前用户可见的评分列表（是否含他人分由项目 review_rule 决定）  
- `POST /api/v1/review/submissions/<id>/scores/`  body: `{ indicator_id, score, comment?, round_type? }`  → 评审教师打分（双评会自动区分 round_type）  
- `POST /api/v1/review/submissions/<id>/arbitrate/`  body: `{ indicator_id, score, comment? }`  → 仲裁打分  
- `POST /api/v1/scoring/import/`  **form-data**：`project_id`, `file`（Excel）  → 批量导入成绩  

**前端要做的**：
- 页面：**审核任务列表**（待办/已办 Tab 或筛选）；**审核详情**（展示提交内容、佐证、当前评分；辅导员有「初审」按钮；评审有「按指标打分」表单，可多指标提交；若需仲裁则显示仲裁入口）；**批量导入**（上传 Excel + 选择项目，展示导入结果/错误信息）。
- Excel 上传用 FormData，同模块4 佐证上传方式。

**设计要点**：审核流程清晰；打分表单与指标一一对应；导入结果可读（成功条数、错误行提示）。

**交付标准**：辅导员能初审、评审能双评、仲裁能录入仲裁分、管理员能导入 Excel，且提交状态与最终分正确更新。

# 模块6：申诉

**目标**：学生对某条提交发起申诉、在待处理时可修改或撤回申诉理由；有权限的角色（辅导员/主任/管理员）可处理申诉（通过/驳回并填处理意见）。毕设要求：一键申诉、可修改或撤回申诉理由。

**后端接口（已存在）**：
- `POST /api/v1/submissions/<id>/appeal/`  body: `{ reason }`  → 提交申诉  
- `GET /api/v1/appeals/`  → 申诉列表（学生只看自己的，管理员看全部）  
- `GET /api/v1/appeals/<id>/`  → 申诉详情  
- `PATCH /api/v1/appeals/<id>/`  body: `{ reason }`  → 修改理由（仅 pending）  
- `POST /api/v1/appeals/<id>/handle/`  body: `{ status: "approved"|"rejected", handle_comment? }`  → 处理申诉  

**前端要做的**：
- 学生端：在「我的提交」详情或单独入口「申诉」：填写理由提交；申诉列表可查看状态并可对 pending 的申诉修改理由或直接撤回（DELETE）。  
- 管理/辅导员端：申诉列表（筛选状态）；点击某条进入详情，有「通过/驳回」按钮并填处理意见，调用 handle 接口。

**设计要点**：申诉入口明显；处理界面有通过/驳回及意见输入。

**交付标准**：学生能提交/修改/撤回申诉，有权限角色能处理并填写意见。

# 模块7：成绩报表

**目标**：学生可查看本人各项目得分与明细；主任/管理员可查看某项目的汇总（按班级等）、排名、并导出 Excel/PDF。毕设要求：总分排名、多维度统计、导出 Excel/PDF。

**后端接口（已存在）**：
- `GET /api/v1/report/student/me/`  → 本人各提交的得分与 score_detail（学生）  
- `GET /api/v1/report/project/<id>/summary/`  → 某项目按班级等汇总（主任/管理员）  
- `GET /api/v1/report/project/<id>/ranking/`  query: `?page=&page_size=`  → 排名分页  
- `GET /api/v1/report/project/<id>/export/?format=xlsx` 或 `format=pdf`  → 导出文件（浏览器会下载）  

**前端要做的**：
- 学生：**我的成绩**页，展示 `report/student/me` 返回的列表（项目、提交、最终分、明细）。  
- 主任/管理员：**项目报表**：选择项目后，展示汇总、排名表（分页）；提供「导出 Excel」「导出 PDF」按钮（打开或跳转 export 接口 URL 即可触发下载）。

**设计要点**：表格清晰；导出按钮明显。

**交付标准**：学生能看自己成绩；管理员能看汇总与排名并成功导出文件。

# 模块8：系统管理与审计

**目标**：管理员可管理补交申请与补交通道、集中处理待推送补交记录、查看审计日志（含异常高亮）、对某条提交做最终分修改（必填理由，留痕）。毕设要求：补交通道可手动开启并按范围控制，支持批量推送给评阅流程；管理员改分必须留痕；异常操作高亮。

**后端接口（已存在）**：
- `GET /api/v1/admin/late-requests/`  → 补交申请列表  
- `POST /api/v1/admin/late-requests/<id>/handle/`  body: `{ action, comment, hours? }`  → 处理申请（批准时可开通道）  
- `GET /api/v1/admin/late-channels/`  → 通道列表（支持 `active_only=1` 与 `include_submissions=1`）  
- `POST /api/v1/admin/late-channels/`  body: `{ scope_type, student_no|username|class_id, project_id?, reason, hours }`  → 手动开启补交通道  
- `POST /api/v1/admin/late-channels/<id>/close/`  body: `{ comment? }`  → 手动关闭通道  
- `POST /api/v1/admin/late-channels/<id>/batch-push/`  body: `{ submission_ids? }`  → 批量推送补交记录至评审  
- `GET /api/v1/audit/logs/`  query: `?is_abnormal=1`, `?action=xxx`  → 审计日志列表  
- `GET /api/v1/audit/my-logs/`  → 当前账号日志列表  
- `POST /api/v1/admin/score-override/`  body: `{ submission_id, final_score, reason }`  → 管理员改分（会写 AuditLog，is_abnormal=True）  

**前端要做的**：
- **补交通道管理**：建议拆为「补交申请」「手动开启通道」「通道列表」「待推送中心」四个子模块。  
- **审计日志**：表格展示日志列表，支持按「是否异常」筛选；异常行可高亮（如红色或加粗）。  
- **改分**：表单或弹窗（选择提交或输入 submission_id、填写新分数、必填理由），提交后调用 score-override。

**设计要点**：管理功能入口清晰；审计表可读；改分表单强调「必填理由」。

**交付标准**：管理员能开启补交通道、查看审计日志（含异常高亮）、对指定提交修改最终分并留痕。

# 如何把「单模块」交给 AI 做（复制用）

做某一模块时，可在新对话里粘贴下面模板，并把「模块X」和「模块名」换成实际编号与名称（如「模块4」「学生端-我的提交」）：

```
项目：校园综合测评管理系统（Django + Vue 3），后端 API 已全部实现，前端是骨架。

请只做「模块X：模块名」的前端对接与页面实现，不要改后端。

具体要求：
```
