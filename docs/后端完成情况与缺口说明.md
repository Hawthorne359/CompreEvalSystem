# 后端完成情况与缺口说明

## 总体结论

**后端主体已全部写好**，规划中的核心接口、模型、权限、业务逻辑（双评、仲裁、总分计算、审计留痕等）都已实现，可以直接被前端按《按模块逐步完善指南》对接。

下面按「已具备」和「小缺口/可选增强」分开说明，便于你决定是否要补。

---

## 一、已具备的后端内容

### 1. 认证与用户（users）
- 登录、刷新、当前用户、角色切换（含切到管理员二次验证）
- 用户列表、详情、创建、更新（含密码）；权限：管理员可管全部，本人可读/改自己

### 2. 组织架构（org）
- 院系列表（支持 tree=1 树形）、班级列表（支持按院系/年级筛选）、某班学生列表  
- **仅只读**：无院系/班级的增删改接口（规划里管理员可在 Django Admin 维护，或后续加）

### 3. 测评项目与规则（eval）
- 测评周期：列表、创建
- 某周期下项目：列表、创建
- 项目：详情、更新
- 指标：列表、创建、详情、更新、删除
- 总分权重规则：获取、更新（get_or_create）
- 双评/仲裁规则：获取、更新（get_or_create）

### 4. 学生提交（submission）
- 提交列表（按项目/状态筛选）、创建草稿、详情、更新草稿、正式提交、上传佐证（form-data）

### 5. 审核与评分（scoring）
- 审核任务列表、某提交审核详情、辅导员初审、评审打分（双评）、仲裁、评分列表（按规则是否可见他人）、Excel 批量导入

### 6. 申诉（appeal）
- 提交申诉、申诉列表、申诉详情、修改/撤回理由（仅 pending）、处理申诉

### 7. 成绩报表（report）
- 学生本人成绩、项目汇总、排名（分页）、导出 Excel/PDF

### 8. 系统管理与审计（audit）
- 补交申请管理（查询、审批）、补交通道管理（手动开启、关闭、列表）、批量推送补交记录、审计日志、管理员改分

### 9. 模型与逻辑
- 规划中的表（User、Role、UserRole、Department、Class、EvalSeason/Project/Indicator、ScoreWeightRule、ReviewRule、StudentSubmission、Evidence、ScoreRecord、ArbitrationRecord、Appeal、OperationLog、LateSubmitRequest、LateSubmitChannel、导入批次等）均已存在
- 总分按 formula_config 加权计算、双评分差超阈值触发仲裁、改分/补交留痕等逻辑已实现

## 二、小缺口与可选增强

以下不影响「按文档把前端做出来」，但若你希望体验更完整，可以后续补。

### 1. 用户角色分配接口（模块 3 用户管理）

- **现状**：创建/编辑用户时，API 只支持「用户基本信息 + 院系/班级」，**没有**「为用户添加/删除角色」的接口。
- **影响**：前端「用户管理」里不能直接给用户勾选「学生/辅导员/评审/主任/管理员」等角色，目前只能到 **Django Admin** 里对 `UserRole` 表操作，或先建用户再在 Admin 里分配角色。
- **可选补法**：增加例如 `POST /api/v1/users/<id>/roles/`（body: `{ role_id, scope_type?, scope_id?, is_primary? }`）、`DELETE /api/v1/users/<id>/roles/<ur_id>/`，由管理员在用户管理页直接分配/移除角色。

### 2. 学生查看「评审意见」（模块 4 学生端）

- **现状**：提交详情接口返回 `status`、`final_score`、`score_detail`、`evidences` 等，**没有**带出每条评分记录里的 `comment`（评审意见）。
- **影响**：学生端「查看评审意见」若只靠当前提交详情，看不到老师写的评语。
- **可选补法**：  
  - 在提交详情序列化里加只读的 `score_records`（或仅 `comment` 列表），只对提交者本人暴露；或  
  - 单独提供 `GET /api/v1/submissions/<id>/review-comments/`，返回当前用户可见的评分与评语。

### 3. 单个测评周期详情（模块 2 测评配置）

- **现状**：只有 `GET /seasons/` 列表和 `GET /seasons/<id>/projects/`，**没有** `GET /seasons/<id>/` 单条周期详情。
- **影响**：若前端在「某周期下的项目列表」页想展示周期名称等信息，需要从列表里已拿到的周期对象里取，或传参；一般够用。
- **可选补法**：增加 `GET /api/v1/seasons/<id>/`，返回单条 EvalSeason。

### 4. 院系/班级的增删改（模块 1 组织架构）

- **现状**：院系、班级只有**只读**接口（列表、班级学生），无 POST/PATCH/DELETE。
- **影响**：组织架构数据只能在 **Django Admin** 里维护，或手工在数据库/SQL 里加。
- **可选补法**：为 Department、Class 增加对应的 Create/Update/Delete 视图与路由，并做权限控制（仅管理员）。

## 三、总结表

| 类别           | 状态说明 |
|----------------|----------|
| 规划中的核心 API | 已全部实现，可直接对接前端 |
| 用户角色分配     | 无专用 API，暂用 Django Admin |
| 学生端评审意见   | 详情中未带 comment，可后续加字段或单独接口 |
| 单条周期详情     | 无，非必须，可从列表取 |
| 院系/班级写操作  | 无，可用 Admin 或后续加接口 |

**结论**：后端**算全部写好**，能支撑《按模块逐步完善指南》里各模块的对接；上面几项是**可选增强**，按你需要再补即可。
